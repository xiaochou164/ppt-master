# PRD - Web Console（已完成部分与当前方案）

## 1. 文档目的

本文记录 `PPT Master` 本地 Web Console 的两类内容：

- 已经完成并落地的部分
- 当前已经明确的产品思路与实现方案

文档原则：

- 不把未开发能力写成已上线能力
- 不把站外 AI 对话流程误写成站内能力
- 对已经确定的思路和方案，明确写清取舍与边界

---

## 2. 产品定位

Web Console 是一个面向本地工作流的浏览器控制台，用来把 `PPT Master` 的项目管理、状态检查、讲稿生成、后处理执行、结果预览与下载集中到一个页面里。

当前定位是：

- 服务本地单机使用
- 辅助串联既有 PPT 生产流程
- 对流程做可视化提示和状态检查
- 把适合网页承接的步骤放进浏览器
- 保持对原有串行流水线约束的遵守

明确不属于当前已完成范围的能力：

- 浏览器内直接完成模板决策
- 浏览器内直接完成 Strategist 八项确认
- 浏览器内直接生成 SVG 页面
- 浏览器内直接执行 AI 配图
- 多用户、权限、登录、云端协作

---

## 3. 总体思路

### 3.1 核心思路

Web Console 不是要替代整个 `PPT Master` AI 生产链路，而是要把“适合网页承接的确定性步骤”收进浏览器，把“仍依赖强 AI 交互与多轮确认的步骤”继续留在 AI 编辑器或终端。

因此当前采用的是一种“半闭环控制台”思路：

- 浏览器负责项目、状态、校验、执行、预览、下载
- AI 编辑器继续负责模板决策、八项确认、页面 SVG 生成、条件性配图
- 两边通过项目目录和文件状态衔接，而不是重新定义一套独立协议

### 3.2 为什么采用这个方案

原因有三点：

- 主流程里有一部分步骤本质上是开放式设计决策，不适合先硬塞进网页表单
- 后处理、校验、讲稿生成、下载预览这类步骤是强确定性的，适合网页承接
- 先做“本地工作流控制台”，可以在不破坏原有技能链路的前提下快速形成可用闭环

### 3.3 当前产品方案

当前 Web 方案不是“全站内生成 PPT”，而是“围绕项目目录的流程控制层”。

具体方案是：

- 用一个本地 HTTP 服务暴露项目、模型配置、后处理和文件访问接口
- 用一个单页前端按流程步骤组织用户操作
- 用文件存在性和目录状态作为阶段推进依据
- 对不能站内完成的步骤，提供明确的状态提示和可直接发给 AI 的起始指令
- 对能站内完成的步骤，直接调用现有脚本并回显日志

## 4. 分阶段承接方案

### 4.1 项目与来源阶段

方案是直接网页化，因为这部分结构稳定、输入清晰、无需复杂 AI 多轮博弈。

网页负责：

- 创建项目
- 选择画布格式
- 导入本地文件、URL、粘贴文本
- 把内容统一落到项目目录

### 4.2 模板决策与 Strategist 阶段

方案是“状态提示 + AI 指令引导”，而不是先做浏览器内表单化替代。

这样做的原因是：

- 模板选择和八项确认高度依赖上下文判断
- 当前最稳定的执行环境仍然是 AI 编辑器对话
- 浏览器先承担流程提醒、状态识别、上下文交接即可

### 4.3 Image_Generator 阶段

方案是条件提示，不在网页里直接跑图。

理由是：

- 是否需要配图依赖 `design_spec.md`
- 图片生成链路、质量控制和人工判断目前仍更适合站外执行
- 当前网页只需要告诉用户“该不该做、什么时候做、看什么文件”

### 4.4 Executor 阶段

方案是拆成两个部分承接：

- `svg_output/*.svg` 继续由 AI 编辑器完成
- `notes/total.md` 由网页在条件满足后调用模型生成

这是当前最关键的整合方案，因为它把 Executor 中最适合网页接管的部分先落地了。

### 4.5 后处理阶段

方案是严格网页化，但继续坚持串行约束。

具体原则：

- 每一步单独触发
- 前后端双重 guard
- 不允许跳过 `Split Notes`
- 不允许绕开 `Finalize SVG` 直接导出
- 导出固定基于 `svg_final/`

### 4.6 交付阶段

方案是“结果总览 + 文件访问 + 预览下载”。

网页不重新解释产物，而是直接把项目中已经生成的关键文件集中呈现出来。

## 5. 已完成目标

### 5.1 流程可视化

已完成一个 7 步式流程界面，覆盖：

1. 创建项目
2. 导入来源
3. 模板决策
4. 策略与设计规范
5. AI 配图（可选）
6. 生成页面与讲稿
7. 后处理与导出

其中第 3、4、5、6 步对”哪些必须在 AI 编辑器/终端完成、哪些已经能在网页里完成”做了明确区分，避免用户误操作。

### 5.2 本地项目工作台

已支持：

- 在 `projects/` 下创建项目
- 展示最近项目
- 按项目名、格式、日期筛选项目
- 读取项目状态并在页面中持续刷新
- 展示项目上下文摘要，包括来源数、SVG 数、PPTX 数、备注拆分状态等

### 5.3 来源导入

已支持通过网页导入以下内容到项目：

- 本地路径
- URL
- 直接粘贴 Markdown
- 直接粘贴纯文本

同时已支持：

- 为粘贴内容生成文件后再导入
- 选择 `move` 模式移动文件而不是复制
- 复用现有 `project_manager.py import_sources` 能力

### 5.4 模板决策与 Strategist 阶段承接

网页端已实现”状态提示 + AI 指令引导”模式，并已新增部分站内执行能力。

已完成：

- 检测模板决策是否已被下游产物隐式完成
- 检测 `design_spec.md` 是否已经存在
- 为模板决策阶段生成可直接发给 AI 的起始指令
- 为 Strategist 阶段生成可直接发给 AI 的起始指令
- 在界面中明确八项确认仍需在 AI 编辑器或终端完成
- **[新增]** 模板浏览与选择功能
- **[新增]** 应用模板到项目 (`apply-template` 接口)
- **[新增]** 内容分析 API (`analyze`)：自动分析来源内容并生成 Strategist 八项确认建议
- **[新增]** 设计规范保存 API (`design-spec`)：将设计规范写入 `design_spec.md`

### 5.5 AI 配图阶段

已完成完整的站内 AI 配图能力：

- 检测 `design_spec.md` 是否已存在
- 检测是否需要 AI 配图
- **[新增]** 图片模型配置管理（独立于文本模型配置）
- **[新增]** 支持 Gemini 和 OpenAI 两种图片生成后端
- **[新增]** 图片生成 API (`generate-image`)：在网页中直接生成配图
- **[新增]** 支持多种宽高比 (1:1, 16:9, 9:16, 4:3, 3:2 等 14 种)
- **[新增]** 支持多种尺寸 (512px, 1K, 2K, 4K)
- **[新增]** 图片生成结果自动保存到项目 `images/` 目录

### 5.6 Executor 阶段承接

已完成完整的站内 SVG 生成与讲稿生成能力：

- 检测 `svg_output/*.svg` 是否已生成
- **[新增]** SVG 页面生成 API (`generate-svg`)：通过 SSE 流式生成 SVG 页面
- **[新增]** 支持分页生成，自动解析 `design_spec.md` 提取页面大纲
- **[新增]** 支持增量生成，跳过已存在的页面
- **[新增]** SVG 重新生成 API (`regenerate-svg`)：支持重新生成指定页面或全部页面
- **[新增]** SVG 删除 API (`delete-svg`)：支持删除指定页面或全部 SVG 文件
- **[新增]** SVG 预览缓存控制：添加时间戳参数防止浏览器缓存问题
- 在 SVG 页面存在后，开放 `notes/total.md` 生成功能
- 通过已选中的文本模型配置调用 `skills/ppt-master/scripts/notes_gen.py`

- 在页面上展示当前激活的模型配置
- 在日志区展示讲稿生成结果、错误信息和执行反馈

已支持：

## 8. 当前产品边界

为了避免认知偏差，当前 Web Console 的边界明确如下：

- 它是”本地流程控制台”，不是完整在线 PPT 生产平台
- 它已经覆盖项目管理、讲稿生成、后处理、预览下载
- **[已实现]** 模板浏览与选择（站内完成）
- **[已实现]** Strategist 八项确认建议生成（通过 LLM API）
- **[已实现]** 设计规范保存（站内完成）
- **[已实现]** AI 配图（站内完成，支持 Gemini/OpenAI）
- **[已实现]** SVG 页面生成（站内完成，通过 LLM API 流式生成）
- 它强调沿用原有串行流程，而不是重写流程规则

## 9. 方案取舍

当前方案背后的取舍是明确的：

- 不追求一步到位做成全在线平台，优先做可运行的本地控制台
- **[已调整]** 开放式 AI 决策流程现已部分表单化：模板选择、设计规范、SVG 生成、AI 配图均可在站内完成
- 不改造底层生产脚本，优先复用既有 `project_manager.py`、`notes_gen.py` 和后处理脚本
- 不为”自动化感”牺牲流程正确性，优先保留串行约束和 guard

这意味着当前版本是”控制台 + 工作流桥接层 + 部分站内生成能力”，而非”完整替代 AI 编辑器的 Web 产品”。

## 10. 已完成定义（Done）

当前 Web Console 已完成的完整闭环是：

1. 用户在网页中创建项目
2. 用户在网页中导入来源内容
3. **[新增]** 用户在网页中浏览并选择模板（可选）
4. **[新增]** 用户在网页中生成 Strategist 八项确认建议（调用 LLM API）
5. **[新增]** 用户在网页中确认并保存设计规范
6. **[新增]** 用户在网页中生成 AI 配图（可选）
7. **[新增]** 用户在网页中流式生成 SVG 页面
8. 用户在网页中配置文本模型并生成 `notes/total.md`
9. 用户在网页中顺序执行 `Split Notes`、`Finalize SVG`、`Export PPTX`
10. 用户在网页中预览 SVG 并下载最终 PPTX

这条链路已经具备可运行、可校验、可交付的完整闭环能力。
